1. Vue d'ensemble du projet

1.1 Objectif

La Cartographie Interactive est une application web collaborative destinee au service commercial et d'estimation des couts. Elle permet de :

  • Visualiser les 14 etapes du processus commercial sous forme de flowchart interactif
  • Organiser hierarchiquement les documents : Entreprise > Projet (AIM-XXXX) > Dossier par etape > Fichier
  • Collaborer en equipe avec un systeme de roles et permissions granulaire (RBAC)
  • Stocker et synchroniser les donnees dans le cloud via Firebase

1.2 Fonctionnalites principales

FonctionnaliteDescription
Flowchart interactif14 etapes cliquables avec description, actions et documents attendus
Explorateur de fichiersNavigation hierarchique Entreprise > Projet > Dossier > Fichier

Gestion des projetsCode AIM-XXXX, statuts (active/won/lost/archived), notes
Upload de fichiersDrag & drop par etape du processus
RBAC3 roles : Commercial, Directeur, Admin avec permissions differenciees
Partage de projetsProprietaire peut partager en mode viewer ou editor

2.3 Flux de donnees

Le flux d'authentification et d'autorisation suit ces etapes :

  1. Login : L'utilisateur clique sur "Se connecter avec Kimi" → redirection vers Kimi OAuth → callback avec code → echange contre JWT token → session creee
  2. Chaque requete : Le client tRPC envoie le JWT dans les headers → le middleware verifie la session → injecte l'utilisateur dans le contexte
  3. Autorisation : Le middleware RBAC verifie le role et les permissions projectMembers avant d'executer la requete

3.3 Relations et contraintes

  • users 1-N companies : Un utilisateur peut creer plusieurs entreprises (createdBy)
  • users 1-N projects : Un utilisateur peut posseder plusieurs projets (ownerId)
  • companies 1-N projects : Une entreprise a plusieurs projets (companyId)
  • projects 1-N projectFiles : Un projet a plusieurs fichiers (projectId)
  • projects N-N users via projectMembers : Un projet peut avoir plusieurs membres, un utilisateur peut acceder a plusieurs projets

5. Systeme de permissions (RBAC)

5.1 Les 3 roles

Le systeme implemente un Role-Based Access Control a 3 niveaux :

Commercial (role par defaut)

  • Visualiser : Ses propres projets + les projets partages avec lui
  • Modifier : Ses propres projets uniquement (lui est owner)
  • Partager : Ses projets uniquement (en tant que owner)
  • Supprimer : Ses projets uniquement

Directeur

  • Visualiser : Tous les projets (vision globale)
  • Modifier : Sur autorisation explicite du proprietaire (permission editor)
  • Partager : Non
  • Supprimer : Non

Admin

  • Tout : Controle complet sur toutes les donnees
  • Peut modifier, supprimer et partager n'importe quel projet
  • Peut modifier les entreprises (update/delete)

5.2 Logique de permission

La logique est implementee dans api/middleware.ts via la fonction getProjectPermission() :

function getProjectPermission(userId, projectId, userRole):
  SI userRole == "admin" :
    RETOURNER "owner"           // Admin = full control

  SI userRole == "directeur" :
    member = chercher dans projectMembers
             WHERE projectId = projectId AND userId = userId
    SI member existe ET member.permission == "editor" :
      RETOURNER "editor"
    SINON :
      RETOURNER "viewer"        // Directeur peut toujours voir

  SI userRole == "commercial" :
    project = chercher projet par projectId
    SI project.ownerId == userId :
      RETOURNER "owner"
    member = chercher dans projectMembers
             WHERE projectId = projectId AND userId = userId
    SI member existe ET member.permission != "pending" :
      RETOURNER member.permission
    RETOURNER null               // Pas d'acces

5.3 Middleware d'autorisation

Quatre niveaux de middleware sont disponibles :

MiddlewareRole requisUsage
publicQueryAucunEndpoints publics (ping)
authedQueryTout utilisateur connecteMajorite des endpoints
directeurQuerydirecteur OU adminModification entreprises
adminQueryadmin uniquementOperations sensibles
Permission dynamique vs statique : Les middlewares statiques (authedQuery, directeurQuery, adminQuery) verifient le role. La permission sur un projet specifique est verifiee dynamiquement dans chaque endpoint via getProjectPermission(), canEditProject() et canViewProject().

7. Guide de modification

7.1 Ajouter un dossier au processus commercial

Les dossiers du processus sont definis dans src/data/flowData.ts. Chaque etape a un champ folderName qui determine son dossier parent.

Etapes :

  1. Ouvrir src/data/flowData.ts
  2. Ajouter ou modifier un objet FlowNode dans le tableau flowNodes
  3. Definir le folderName pour grouper les etapes par dossier
  4. Si une nouvelle etape est ajoutee, ajoutez aussi son icone dans Flowchart.tsx
// Exemple : ajouter une nouvelle etape "Analyse concurrentielle"
{
  id: 'analyse-concu',
  step: 5,
  title: 'Analyse concurrentielle',
  type: 'step',
  folderName: '02 - DOSSIER ETUDE',  // Groupe avec les etudes
  description: 'Etude des offres concurrentes...',
  details: ['Lister les concurrents', 'Analyser les prix', '...'],
  documentTypes: ['Benchmark concurrentiel'],
  icon: 'search'
}
Important : Le folderName est un simple texte. Les etapes avec le meme folderName sont automatiquement groupees dans l'Explorateur de Fichiers. Pas besoin de creer le dossier explicitement.

7.2 Modifier les permissions

Le RBAC est centralise dans api/middleware.ts.

Ajouter un nouveau role :

  1. Modifier l'enum role dans db/schema.ts
  2. Ajouter le role dans requireRole() dans api/middleware.ts
  3. Mettre a jour la logique getProjectPermission() si necessaire
  4. Executer npm run db:push pour mettre a jour la DB

Modifier les permissions d'un role existant :

Editez la fonction getProjectPermission() dans api/middleware.ts. Par exemple, pour donner aux Directeurs le droit de modifier sans autorisation :

// Avant (requiert autorisation)
if (userRole === "directeur") {
  const member = await db...
  if (member.length > 0 && member[0].permission === "editor") {
    return "editor";
  }
  return "viewer";
}

// Apres (modification libre)
if (userRole === "directeur") {
  return "editor";  // Directeur peut toujours editer
}

7.3 Ajouter un endpoint API

Pour ajouter un nouvel endpoint tRPC :

Etape 1 : Ajouter la logique dans le routeur

Dans api/projects-router.ts (ou le routeur approprie) :

myNewEndpoint: authedQuery
  .input(z.object({ projectId: z.number() }))
  .query(async ({ ctx, input }) => {
    // Verifier les permissions
    const hasAccess = await canViewProject(ctx.user.id, input.projectId, ctx.user.role);
    if (!hasAccess) throw new TRPCError({ code: "FORBIDDEN" });

    // Logique metier
    const result = await db.select().from(...)...;
    return result;
  }),

Etape 2 : Creer le hook frontend

Dans src/hooks/useProjects.ts :

export function useMyNewEndpoint(projectId: number) {
  return trpc.projects.myNewEndpoint.useQuery({ projectId });
}

Etape 3 : Utiliser dans un composant

const { data, isLoading } = useMyNewEndpoint(projectId);

7.4 Modifier le frontend

Ajouter une nouvelle page :

  1. Creer le composant dans src/pages/NouvellePage.tsx
  2. Ajouter la route dans src/App.tsx
  3. Utiliser les hooks tRPC existants pour charger les donnees

Modifier les styles globaux :

Editez src/index.css. Les styles Tailwind sont deja configures. Vous pouvez ajouter des classes CSS personnalisees dans la section @layer components.

Ajouter un composant shadcn/ui :

npx shadcn add <component-name>
# Exemple : npx shadcn add dialog
